1. Project Overview

Univu is a full-stack astrology platform offering personalised Vedic, Western, Chinese, Egyptian, and Mayan readings powered by a 4-tier AI fallback chain and a local astronomy engine. It is a Next.js 14 App Router app deployed on Vercel, running entirely within a free-tier budget.

2. Core Goals

  • Accuracy first — Planetary positions computed via astronomy-engine v2.1.19 (VSOP87, arcsecond accuracy). No pre-stored ephemeris data.
  • AI-driven personalisation — 12 distinct reading sections per birth chart, each with a specific Vedic-trained prompt. All responses are chart-specific, never generic.
  • Free-tier economics — Gemini (free quota) → Cerebras (free) → Groq ($0.59/1M) → OpenRouter free models. Monthly AI cost: effectively $0 for typical traffic.
  • Privacy by design — Real name never reaches any AI model. Name is replaced with "the Seeker" before any API call. No user data persisted.
  • Internationalisation — 19 languages, 11 Indic scripts for Panchangam, locale auto-detected from browser.
  • Accessibility — Age gate, consent gate, disclaimers on all AI-generated content, PII protection.

3. Data Flow — Birth Chart Reading

User fills IntakeForm (12 fields)
  │
  ▼
POST /api/reading
  ├─ apiGuard(req)  → rate limit 100/min/IP, origin check, sanitise()
  ├─ Parse: name, dob, timeOfBirth, birthLat/Lng, currentLat/Lng, gender, tradition…
  ├─ julianDay(dob, timeOfBirth, birthLng)   ← local→UTC conversion
  ├─ lahiriAyanamsa(year)                    ← 23.856° + drift
  ├─ SunPosition(birthDateUTC)               ← astronomy-engine (VSOP87)
  ├─ EclipticGeoMoon(birthDateUTC)           ← astronomy-engine
  ├─ GeoVector(Mercury/Venus/Mars/Jupiter/Saturn, birthDateUTC)  ← astronomy-engine
  ├─ calculateLagna(jd, T, lat, lng, ayanamsa)
  ├─ calculatePanchanga(sunSidereal, moonSidereal, jd)
  ├─ calculateDashaTimeline(moonSidereal, birthYear)
  ├─ checkManglik(planets, moonRashiIdx)     ← 7-rule BPHS cancellations
  └─ return FullReading (50+ fields)

Client receives FullReading
  ▼
ReportCard.tsx renders 10 tabs
  ├─ Each AI tab: fetch /api/personal-reading { section, context }
  │     ├─ budgetGuard()
  │     ├─ buildSectionPrompt(section, context)  ← chart-specific prompt
  │     └─ callAI() → 4-tier fallback → return { text, provider, tokensUsed }
  └─ StoryAnimator: fetch /api/enrich-story per scene (mythic-anime voice)

3. Chat System Prompt Architecture

The chat route injects chart context when a reading has been generated. Dasha-first timing is enforced via rules 10–14 in the system prompt.

System prompt = ASTRO_SYSTEM base (13 rules)
  + IF chartContext:
      "IMPORTANT — The user's ACTUAL chart data:
       {lagnaSign, moonSign, sunSign, currentDasha, antardasha, concern}
       Rules 14 additional: cite exact houses, dasha scan, NEVER invent,
       Timing method: antardasha scan in 3-year blocks from NOW"
  + Current year injected: ${new Date().getFullYear()}
  + Hard limits: never predict marriage >45, never give timing >7yr window only

1. Security Architecture

Request Flow:

Browser
  ↓
Vercel Edge Middleware (middleware.ts)
  ├── MAINTENANCE=true → redirect /maintenance (except localhost)
  └── Pass through

Next.js API Route Handler
  ↓
apiGuard(req)
  ├── Rate limit: 100 req/IP/min (in-memory Map) → 429 Retry-After:60
  ├── Origin check: univu.vercel.app | *.vercel.app | localhost → else 403
  ├── Request size check → 413 if oversized
  └── sanitise(value, maxLen) — strips HTML tags, injection chars on all string inputs

budgetGuard()  [AI routes only]
  ├── Check getStats().isOverBudget
  └── $100/month cap, per-provider rates; FORCE_SHUTDOWN env override

PII protection
  └── name replaced with "the Seeker" in ALL AI prompts before any API call
